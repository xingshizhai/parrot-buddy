# 鹦鹉叫声识别自动优化闭环方案（AutoResearch 风格）

## 1. 目标与原则

目标不是“人工手调参数”，而是构建一个持续运行的自动研究系统：
- 自动提出改进假设
- 自动修改代码/参数
- 自动编译、烧录、回放测试集
- 自动评估与回归分析
- 仅在指标提升且满足安全门禁时合入

核心思想参考 AutoResearch：
- 让系统自己提出实验并执行
- 让结果驱动下一轮实验
- 用排行榜和可复现实验记录替代主观判断

---

## 2. 总体架构（多设备协同）

```
┌────────────────────────────────────────────────────────────┐
│                    Orchestrator Host (主机)                │
│  Python Runner + Experiment DB + Artifact Store + Scheduler │
└───────────────┬──────────────────────────┬─────────────────┘
                │                          │
                │(生成候选修改)             │(运行实验)
                ▼                          ▼
      ┌──────────────────┐        ┌─────────────────────────┐
      │ LLM Services     │        │ HIL Test Farm           │
      │ - 代码修改建议    │        │ - ESP32 DUT(被测板)      │
      │ - 实验计划生成    │        │ - 1~N 外部ESP32回放板     │
      │ - 结果总结        │        │ - USB继电器/电源控制      │
      └──────────────────┘        │ - 麦克风/音频回采设备      │
                                  └──────────┬──────────────┘
                                             │
                                             ▼
                                  ┌─────────────────────────┐
                                  │ Metrics + Judge         │
                                  │ - 误触发/漏检            │
                                  │ - 延迟/稳定性            │
                                  │ - 多目标评分             │
                                  └─────────────────────────┘
```

角色定义：
- Orchestrator Host：唯一真相源，负责任务调度与决策。
- LLM Services：生成候选 patch 与实验假设，不直接改主分支。
- DUT（Device Under Test）：运行待评估固件。
- Playback Boards：外部 ESP32 板卡，模拟真实场景声音回放触发。
- Judge：统一计算指标，执行门禁策略。

---

## 3. 自动优化闭环（Test -> Modify -> Retest）

每轮循环（一个 Trial）包含：

1. 生成候选
- 输入：当前最佳版本 + 近期失败案例 + 指标短板。
- LLM 输出：
  - 参数修改（阈值、帧数、冷却时间等）
  - 规则结构修改（频带特征、双阈值、抑制器）
  - 代码补丁（受限于白名单文件）

2. 静态门禁
- `idf.py build` 必须通过。
- clang-tidy/cppcheck（可选）无高危问题。
- 禁止修改白名单外文件（例如只允许 `parrot_core`、测试脚本、Kconfig）。

3. 自动部署
- 烧录 DUT 固件。
- 回放板自动播放测试音频集（多场景、多SNR、多音量）。
- 通过串口日志和回采音频收集结果。

4. 自动评估
- 计算分类指标：Precision/Recall/F1。
- 计算产品指标：每小时误触发、触发延迟、CPU/内存占用、温升/重启率。
- 输出综合分数。

5. 决策
- 若优于当前基线并满足硬门槛，进入候选池。
- 若连续 N 轮无改进，触发策略切换（例如从参数搜索切到结构搜索）。

---

## 4. 关键指标与评分函数

建议同时使用“硬门槛 + 多目标分数”：

硬门槛（必须满足）：
- 漏检率 <= 目标阈值
- 每小时误触发 <= 目标阈值
- 平均触发延迟 <= 目标阈值
- 无崩溃、无看门狗复位

综合分数示例：

$$
Score = w_1 \cdot F1 - w_2 \cdot FP_{hour} - w_3 \cdot Latency_{ms} - w_4 \cdot CPU_{pct} - w_5 \cdot Mem_{kb}
$$

其中权重 $w_i$ 可按产品阶段调整：
- 早期更重 Recall（避免漏检）
- 上线前更重 FP/hour（控制误触发体验）

## 5. 测试资产体系（数据与场景）

## 5.1 数据分层
- `train_pool/`：用于模型或规则探索（可被用于自动搜索）。
- `dev_set/`：快速迭代评估，允许频繁使用。
- `holdout_set/`：盲测集，仅用于晋级评估，防止过拟合优化流程。

## 5.2 场景矩阵
至少覆盖：
- 环境噪声：安静/电视/音乐/多人说话/风扇
- 距离：0.5m/1m/2m
- 音量：低/中/高
- 设备差异：不同麦克风灵敏度、不同箱体摆位

## 5.3 自动标注对齐
- 回放板发送“片段开始/结束”时间戳到主机。
- DUT 日志输出“触发事件”时间戳。
- Judge 自动对齐并判定 TP/FP/FN。

## 6. 外部设备接入方案

## 6.1 外部主机
- 建议 Linux 小主机（NUC/工控机）长期运行调度器。
- 服务组件：
  - Experiment Queue（Redis/SQLite）
  - Artifact Store（MinIO/本地磁盘）
  - Dashboard（Grafana/Streamlit）

## 6.2 外部 LLM 服务
- 可同时接入多个 LLM（主模型 + 审核模型）：
  - 生成 patch（主模型）
  - 风险审查与回归点检查（审核模型）
- 采用“结构化输出协议”（JSON）：
  - 修改文件列表
  - 变更理由
  - 预期提升指标
  - 回滚策略

局域网 Ollama 接入建议（免费优先）：
- 在局域网其他主机部署 Ollama（例如 `http://<LAN_IP>:11434`）。
- Orchestrator 使用可配置模型路由：
  - `proposer_model`：负责提出参数/代码候选
  - `critic_model`：负责审查风险、检查回归点
- 采用“本地优先、云端兜底”策略：
  - 默认仅调用局域网 Ollama
  - 当局域网模型不可用时，可选择暂停 trial 或切换云端模型
- 每次调用必须记录：
  - 模型名、主机地址、提示词摘要、响应摘要、耗时、token 统计（若可得）

## 6.3 外部 ESP32 板卡
- 角色一：音频回放器（I2S DAC/PWM+功放）
- 角色二：环境噪声模拟器（可并行）
- 角色三：控制器（继电器切换扬声器、距离工装）

建议最小 HIL 组合：
- 1 台 DUT（ESP32-S3-BOX-3）
- 2 台回放板（“鸟叫”与“干扰源”分离）
- 1 个 USB 继电器（电源循环）

## 7. 修改策略（自动搜索空间）

## 7.1 第 1 层：参数搜索（低风险）
搜索范围示例：
- `PARROT_DETECT_SENSITIVITY`
- `PARROT_TRIGGER_HOLD_FRAMES`
- `PARROT_MIN_TRIGGER_RMS`
- `PARROT_REPLY_COOLDOWN_MS`

方法：
- Bayesian Optimization / TPE
- Hyperband（快速淘汰差方案）

## 7.2 第 2 层：规则结构搜索（中风险）
- 增加频带能量比、ZCR、包络峰宽特征
- 引入双阈值策略（进入阈值/退出阈值）
- 增加噪声自适应抑制器

方法：
- LLM 生成候选规则变体
- 通过模板化代码骨架自动插拔

## 7.3 第 3 层：TinyML 搜索（高收益）
- 小模型架构搜索（DS-CNN 深度、通道数）
- 量化策略搜索（int8/混合量化）
- 级联策略（规则前级 + 模型后级）

---

## 8. 安全与质量门禁（必须自动化）

1. 代码白名单
- 自动优化只允许改动指定路径：
  - `components/parrot_core/**`
  - `main/Kconfig.projbuild`（限定参数区）
  - `tools/autotune/**`

2. 风险检查
- 禁止新增阻塞调用到音频回调路径
- 禁止扩大关键任务栈到阈值以上
- 禁止引入未审计外部依赖

3. 回滚机制
- 每个 Trial 都有唯一 ID 与固件产物
- 若线上验证失败，自动回滚到上一稳定版本

---

## 9. 实验管理与可追溯性

每个 Trial 必须记录：
- 基线 commit / 候选 commit
- 修改 diff
- 测试数据版本
- 硬件拓扑（板卡编号、麦克风编号）
- 指标结果与日志链接
- LLM 提示词与响应摘要

推荐输出：
- 每日 Top-K 候选排行榜
- 周报：指标趋势、失败原因聚类、下周搜索重点

## 11. 分阶段落地计划

### Phase A（1-2 周）
- 搭建 HIL 基础链路：自动烧录 + 回放 + 日志采集
- 先实现参数搜索闭环（不改代码结构）

### Phase B（2-4 周）
- 接入 LLM 生成规则补丁
- 建立白名单与风险门禁
- 形成每日自动优化批次（夜间运行）

### Phase C（4-8 周）
- 引入 TinyML 后级复核
- 实现两级级联自动搜索
- 建立 holdout 盲测晋级流程

---

## 12. 最终建议

推荐采用“参数搜索 -> 规则结构搜索 -> 两级级联 TinyML”的三级路线，配合 HIL 测试农场和 LLM 代码候选生成：
- 自动化程度高
- 可持续优化
- 风险可控
- 与当前 ESP32-S3 工程形态兼容

这套方案的关键不是一次性找到最优参数，而是建立一个能持续产出改进的自动研究系统。

## 13. 真实“播放到麦克风”闭环怎么实现（可直接落地）

这里的关键是构建真实声学链路，而不是数字注入：

`音频文件 -> 外部扬声器播放 -> 空气传播 -> DUT 麦克风采集 -> 算法识别 -> DUT 串口上报结果`

## 13.1 最小可行硬件（MVP）

- 1 台 DUT：ESP32-S3-BOX-3（运行待测固件）
- 1 台播放控制器：任意 ESP32 开发板（建议 S3）
- 1 套扬声器链路：I2S DAC/功放模块 + 小音箱
- 1 个 USB 声卡或测量麦（接主机，用于回采环境声音，非必须但强烈建议）
- 1 个 USB 继电器（可选，用于 DUT 自动断电重启）
- 固定支架/治具：保证 DUT 与扬声器距离、角度可重复

建议固定参数：
- 距离：先固定 40 cm
- 角度：扬声器正对 DUT 麦克风
- 音量：用 1 kHz 校准音定标到固定 SPL（例如 65 dB@DUT 位置）

PC 播放优先建议（当前阶段）：
- 第一阶段直接用主机声卡播放测试音频，不额外引入播放板固件开发。
- 主机同时承担：
  - 音频播放（播放器）
  - DUT 串口采集（识别事件）
  - 回采音频录制（可选）
- 等闭环稳定后，再增加 ESP32 播放节点作为“第二验证通道”。

## 13.2 自动化控制链

主机统一调度三条控制通道：

1. DUT 串口通道
- 下发开始测试命令（可选）
- 采集识别事件日志（触发时间、置信度、阈值）

2. PC 播放控制通道（Phase A）
- 主机本地播放器播放指定 wav 片段
- 播放器输出开始/结束时间戳
- 记录当前输出设备、系统音量、样本率

3. 播放板控制通道（Phase B，可选增强）
- 下发“播放某个 wav 片段”命令
- 播放板返回开始播放时间戳与结束时间戳

4. 回采通道（USB 声卡）
- 主机同步录制环境声，作为事后审计证据

## 13.3 时间同步与标注对齐（重点）

为避免“播放时间”和“识别时间”对不齐，推荐双重对齐：

1. 协议时间戳
- 主机发播放指令时记 `T_host_cmd`
- 播放板实际开始播放回传 `T_play_start`
- DUT 触发日志记录 `T_dut_trigger`

2. 声学锚点
- 每段测试音频前加一个短锚点（如 30 ms chirp）
- 在回采音频中检测 chirp，得到真实声学到达时刻

用声学锚点做最终真值对齐，协议时间戳做辅助诊断。

## 13.4 单轮 Trial 执行流程

1. 烧录 DUT 固件
2. DUT 上电稳定 5-10 秒
3. 按测试计划依次播放样本（鹦鹉叫声、干扰声、静音）
4. 主机同时采集：
- DUT 串口日志
- 播放事件日志
- 回采音频文件
5. Judge 自动计算：
- TP/FP/FN
- 延迟（从声学锚点到 DUT 触发）
- 每小时误触发率（按场景时长归一）
6. 生成 trial_report.json 并入库

## 13.5 关键工程细节

1. 声学环境控制
- 建议使用简易消反射箱体或固定测试角落
- 背景噪声需要记录并纳入报告（dBA）

2. 音量校准
- 每天第一轮先播放校准音
- 若 SPL 漂移超阈值（例如 > 1.5 dB），本轮结果标记为无效

3. 设备漂移与健康检查
- DUT 温度、重启次数、任务堆栈水位纳入 trial 元数据
- 出现异常自动重跑一次并标记 flaky

4. 防止“过拟合测试治具”
- 至少准备两套物理摆位模板（位置 A/B）
- 每轮随机抽取模板，防止算法只适配单一角度

## 13.6 判分与门禁建议

硬门禁（任一不满足即拒绝）：
- 编译通过
- 无 crash/reset
- 误触发率不劣于基线

晋级门禁（建议）：
- 在 dev_set 提升后，必须在 holdout_set 也提升
- 至少连续 3 轮重复实验结果稳定（方差在阈值内）

## 13.7 先做这 3 件事（本周可完成）

1. 搭建一条可复现的声学链路
- 固定距离、固定音量、固定 PC 播放设备

2. 打通自动日志链路
- 主机能同时拿到播放时间、DUT 触发日志、回采音频

3. 先跑纯参数自动搜索
- 只搜索 `PARROT_*` 参数，不改代码结构
- 每晚自动跑 50-100 个 trial，次日输出 Top-10

## 14. 局域网 Ollama 在闭环中的具体用法

## 14.1 推荐角色分工
- `Ollama-1`（proposer）：根据失败样本和历史试验提出新候选。
- `Ollama-2`（critic）：审查 proposer 的修改，输出风险点和拒绝理由。

若只有一台 Ollama 主机，也可以同一模型分两次调用，但提示词模板必须区分“提出者”和“审查者”。

## 14.2 请求/响应规范（建议）
- 请求统一为 JSON 输入：
  - `baseline_metrics`
  - `recent_fail_cases`
  - `allowed_files`
  - `search_budget`
- 响应统一为 JSON 输出：
  - `proposed_edits`（文件+patch）
  - `expected_gain`
  - `risk_notes`
  - `self_confidence`

这样可以避免自然语言输出难以自动解析的问题。

## 14.3 失败回退策略
- Ollama 超时/不可达：
  - 本轮切换到“参数搜索模式”（不做代码修改）
  - 或重试 2 次后标记 `llm_unavailable`
- Ollama 输出不合法 JSON：
  - 自动拒绝并记录样本，进入下一 trial

## 14.4 成本与吞吐优化
- 优先使用局域网模型完成 80% 候选生成。
- 仅对 Top-K 高价值 trial 使用更强模型复审（如果有云端额度）。
- 对重复问题启用响应缓存（相同输入哈希直接复用结果）。
